# BecauseChat 问数能力文档

> 本文档整理项目内**自然语言问数（Text-to-SQL）及相关分析**的能力边界、工具入口与版本差异。  
> 更新日期：2026-03  
> 关联文档：[BeCause2.0能力文档.md](./BeCause2.0能力文档.md)（波动归因与 2.0 算法细节）

---

## 一、总览

问数能力在 BecauseChat 中通过**多类插件工具**组合实现，大致分为三层：

| 层级 | 作用 | 典型插件 Key |
|------|------|----------------|
| **提示词与流程模板** | 从 `.because` 读取命令模板、System/User 提示词，指导 Agent 如何拉 RAG、填变量、调 LLM | `because` |
| **问数技能集（编排执行）** | 统一入口：意图、RAG、Schema、校验、执行、分析、图表等**可调用子能力** | `because_skills`（1.0）、`because_skills_2`（2.0） |
| **直连数据与说明** | 只读 SQL 执行 + 结构化「结论来自哪张表、哪些字段」的说明 | `sql_executor` |

**配置位置**：各工具在 `api/app/clients/tools/manifest.json` 中注册，由 `handleTools.js` 加载对应实现类。

---

## 二、BeCause（`because`）— 提示词模板系统

**定位**：不直接执行 SQL，而是返回**命令说明 + 模板加载指引**，供上层 Agent 按步骤调用 RAG、拼装 Prompt、再让 LLM 生成 SQL。

**主要子命令（节选）**：

| Command | 说明 |
|---------|------|
| `sql-generation` | 生成 SQL（配合 system/user 模板） |
| `sql-generation-reasoning` | 生成推理计划 |
| `intent-classification` | 意图分类 |
| `data-assistance` | 数据辅助 |
| `misleading-assistance` | 误导性查询处理 |
| `scoring` | 文档评分 |
| `hypothetical-questions` | 假设问题生成 |
| `agentic/main-agent`、`agentic/text-to-sql` 等 | Agentic 模式下的主代理与文本转 SQL 等 |

**实现要点**（见 `api/app/clients/tools/structured/BeCause.js` + `BeCauseService`）：

- 命令与 Prompt 模板来自项目根目录 **`.because`**（如 `templates/prompt-templates/`）。
- 说明中强调：**先 RAG**（语义模型、QA、同义词、业务知识等变量），再 Jinja2 类变量替换，最后调用 LLM。

---

## 四、BeCause 问数 2.0（`because_skills_2`）

**定位**：在 1.0 能力集之上增强，实现类为 `BeCauseSkills2`（`Because-2.0` 包）。**新增并强化**波动归因与部分分析链路。

### 4.1 与 1.0 的差异（摘要）

| 维度 | 1.0 (`because_skills`) | 2.0 (`because_skills_2`) |
|------|-------------------------|---------------------------|
| 波动归因 | 无 | **`fluctuation-attribution`**：维度/指标归因、时间对比、下钻 |
| `sql-validation` | 基础校验 | **7 类关键字**、复杂度、双盲对比、文本-知识-SQL 对齐等增强 |
| `result-analysis` | 结果分析 | **Adtributor 相关**、异常检测、趋势、指标关联等增强 |
| `database-schema` | 有 | 有（2.0 实现路径在 Because-2.0） |
| `chart-generation` | 有 | 有（可与 `sql-executor` 联动取数） |

### 4.2 2.0 独有子命令

| command | 说明 |
|---------|------|
| `fluctuation-attribution` | 指标波动根因：维度归因、指标归因、同比/环比/自定义周期、多路径下钻等 |

**算法与模块、输入输出示例、工作流**详见 **[BeCause2.0能力文档.md](./BeCause2.0能力文档.md)**。

## 五、SQL Executor（`sql_executor`）— 独立插件

**定位**：与 `because_skills*` 内嵌的 `sql-executor` 并列存在的**独立工具**，便于只挂「执行 + 归因说明」场景。

**能力**（见 manifest 描述）：

- 调用内部 **hsp-sql-api** 的只读查询能力（如 `/sql_query`）。
- 返回数据行、行数及**结构化归因说明**（表、字段、过滤条件等），便于模型在回答中说明依据。

**环境**：常通过 `SQL_API_URL` 等配置与 SQL 服务地址对齐（与 `handleTools` 中 `database_schema` / `sql_executor` 选项一致）。

## 六、相关能力（扩展）

| 插件 Key | 与问数的关系 |
|----------|----------------|
| `semantic_model_generator` | 语义模型文档生成；可配合 `database_schema`、`sql_executor` 查库、对齐结构 |
| RAG / 知识库 | 在 BeCause 与各 `rag-retrieval` 流程中提供业务语义与样例，**不替代** SQL 执行与校验 |

---

## 七、典型端到端流程（概念）

1. **理解意图**：`intent-classification`（或通过 BeCause 模板做分类/生成）。
2. **上下文**：`rag-retrieval`、必要时 `database-schema`。
3. **生成 SQL**：Agent 使用 BeCause `sql-generation` 类模板 + LLM，或直接推理出 SQL。
4. **校验**：`sql-validation`。
5. **执行**：`sql-executor`（技能集内）或独立 `sql_executor`。
6. **分析与展示**：`result-analysis`；若问「为什么变化」→ **`fluctuation-attribution`（仅 2.0）**；需要图 → `chart-generation`。

## 九、版本与选型建议

- **仅需模板化 Text-to-SQL 流程**：启用 **`because`**，并配置 `.because` 与 RAG。
- **需要一体化技能编排、但不要 2.0 归因**：**`because_skills`**。
- **需要波动归因、增强校验与分析**：**`because_skills_2`**。
- **只要执行 SQL + 结构化归因说明**：可单独启用 **`sql_executor`**，或与上述组合使用。
